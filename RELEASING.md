# Releasing

`tui-components` is a Go library consumed via `go get github.com/RenseiAI/tui-components`, so a "release" is simply a tagged commit on `main`. The workflow in `.github/workflows/release.yml` re-runs the CI validation gate, publishes a GitHub Release with generated notes, and verifies the module is resolvable on `proxy.golang.org`.

## Semver policy

Per `CLAUDE.md`: **no breaking changes within a minor version.**

For this library, a change is breaking if it does any of the following in the public API of `theme/`, `format/`, `component/`, `widget/` (including `widget/notification/`):

- Removes or renames an exported symbol (type, function, method, constant, variable).
- Changes the signature of an exported function or method (parameters, return types, receiver).
- Changes the exported fields of a struct (rename, remove, type change, or adding a required field to a struct that consumers construct by name without `...` zero values).
- Changes the behavior of an exported function in a way a consumer could observe (different output format, different error semantics, different visual rendering where rendering is part of the contract).

Anything internal (under `internal/` or unexported identifiers) is not part of the API contract and may change freely.

Version bump rules:

- Any breaking change in the above list: **major** bump (`X+1.0.0`).
- Any new user-visible feature (`feat:`) without breakage: **minor** bump (`X.Y+1.0`).
- Otherwise (`fix:`, `perf:`, `docs:`, `chore:` user-visible): **patch** bump (`X.Y.Z+1`).

While the module is at `v0.x.y`, minor bumps may carry breaking changes per Go module convention. This is still discouraged — prefer deprecate-then-remove across two minors.

## Conventional commits

The repo uses conventional-commit prefixes. Release notes are generated from them by `git-cliff` (see `cliff.toml`), so the prefix determines which section a commit lands in.

Recognized prefixes:

- `feat:` — user-visible new feature.
- `fix:` — user-visible bug fix.
- `perf:` — performance improvement with no behavioral change.
- `docs:` — documentation only.
- `chore:` — housekeeping (tooling, CI, formatting).

Rules:

- Pick the prefix that matches the **user-visible intent**, not the file touched. A doc-only fix to a misleading godoc example is `fix:` if it changes what consumers think the API does, `docs:` otherwise.
- Scope is optional: `feat(widget): ...` is fine, so is `feat: ...`.
- `chore(deps):` and merge commits are excluded from user-facing release notes by `cliff.toml`.
- Breaking changes: add `!` after the type (`feat(widget)!: ...`) or a `BREAKING CHANGE:` trailer in the body. This flags the change for reviewers and release-notes readers; it does not by itself choose the version bump — you still pick the tag.

## Cutting a release

1. Confirm `main` is green — CI must be passing on the exact commit you are about to tag. Check the Actions tab.
2. Decide the version bump using the rules above. Look at `git log <last-tag>..HEAD` and classify the commits.
3. Create an annotated, signed tag at the tip of `main`:

   ```bash
   git checkout main
   git pull --ff-only
   git tag -s v<X.Y.Z> -m "v<X.Y.Z>"
   ```

   If the team has not adopted signed tags yet, use `git tag -a v<X.Y.Z> -m "v<X.Y.Z>"`. **Flag this to maintainers** — signed tags are the intended policy and should be adopted before the first `v1.0.0`.
4. Push the tag:

   ```bash
   git push origin v<X.Y.Z>
   ```
5. Watch the Release workflow run (`.github/workflows/release.yml` → "Release"). Verify:
   - Validation passes (build, test with `-race`, lint, vuln scan).
   - The GitHub Release appears at the tag with notes generated by `git-cliff`.
   - The `Verify module on proxy.golang.org` step passes.

Do not delete or move a tag once pushed. See Rollback.

## Pre-releases

Convention: `v<X.Y.Z>-rc.<N>` (for example, `v0.1.0-rc.1`). These follow semver's pre-release precedence and are treated as "less than" the equivalent final release by Go's module resolver.

The workflow's tag filter is `v*.*.*` (see `.github/workflows/release.yml`). GitHub Actions tag filters use glob matching where `*` matches any character except `/`, so `v<X.Y.Z>-rc.<N>` tags (for example `v0.1.0-rc.1`) match the filter and trigger the Release workflow — the third `*` absorbs the `-rc.N` suffix. A pre-release can therefore be used as a smoke test of the workflow itself before cutting the corresponding final tag.

Do not rely on the Go module proxy treating a retracted pre-release the same as a retracted final release; if a pre-release is broken, cut `v<X.Y.Z>-rc.<N+1>` rather than retracting.

## Rollback

Tags published to `proxy.golang.org` are **immutable**. The proxy caches modules by version forever, so you cannot safely fix a bad release by re-tagging. To recover:

1. Cut a new patch version with the fix: `v<X.Y.Z+1>`. Follow the normal release steps above.
2. Add a `retract` directive to `go.mod` marking the bad version as withdrawn:

   ```
   retract v<bad-version> // short reason, e.g. "panics on empty input"
   ```

   See [go.dev/ref/mod#go-mod-file-retract](https://go.dev/ref/mod#go-mod-file-retract). Consumers running `go get` will see the retraction and be nudged to a newer version.
3. Optionally delete the *GitHub Release* (not the tag) if it's actively misleading users. The underlying git tag must stay; `proxy.golang.org` has already fetched it and deleting the tag locally or on GitHub does not remove it from the proxy.

## Proxy cache warming

The release workflow pre-warms the Go module proxy so downstream `go get` is fast from the first call. After creating the GitHub Release and before the hard-fail `.info` verification step, the workflow GETs the `list`, `.mod`, `.zip`, and `sum.golang.org/lookup` endpoints. These warm requests are best-effort (`continue-on-error: true`) — a failure does not block the release.

## Troubleshooting

**Lint or tests fail in the release workflow.** Same gates as `ci.yml`; fix on `main`, then tag a *later* commit. Do not reuse the version number — the proxy has already attempted to resolve it and caching can produce confusing results.

**Proxy verification fails.** `proxy.golang.org` can take a few minutes to pick up a new tag. The `Verify module on proxy.golang.org` step retries 5 times with 10, 20, 30, 40, 50-second backoffs. If it still fails after that:

- Confirm the repo is public (the proxy only indexes public modules).
- Confirm the tag is an annotated tag (`git cat-file -t v<X.Y.Z>` should print `tag`, not `commit`). Lightweight tags are not recognized by the proxy for module resolution.
- Re-run the failed workflow job from the Actions UI; the retry loop will run again.

**Release notes are empty or wrong.** `git-cliff` reads `cliff.toml` and the commit range since the last tag. If notes look wrong, check that commits use the recognized prefixes above and that `cliff.toml` has not been changed without intent.
